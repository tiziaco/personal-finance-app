## Summary

Phase 1 builds the invisible infrastructure that every subsequent phase consumes: typed TypeScript interfaces that exactly mirror the Pydantic backend schemas, a shared authenticated fetch utility, React Query hooks for every data domain, and three shared display primitive families (chart wrappers, skeletons, error boundaries). None of these deliverables are user-visible pages — they are the foundation layer consumed silently by Phases 2-7.

The backend schemas are already finalized and inspectable. `TransactionResponse`, `TransactionListResponse`, `AnalyticsResponse`, `DashboardResponse`, `InsightsResponse`, and the full `CategoryEnum` are all defined in `server/app/schemas/`. The frontend already has `@tanstack/react-query` v5 installed and `QueryProvider` properly configured with `useState` (not module-scope). Neither `react-charts` nor `recharts` are installed yet — chart components must be installed and wrapped before any chart renders in Phase 2.

The most important architectural constraint is the **chart library split**: pie charts use shadcn/ui Chart (Recharts) via `npx shadcn add chart`; line and bar charts use `react-charts@beta` wrapped in `next/dynamic` with `ssr: false`. This is locked in STATE.md as a pre-roadmap decision to avoid SSR/dynamic-import complexity. The STATE.md also records an open question about whether `react-error-boundary` is acceptable to add as a package — this must be resolved before PLAN.md is written.

**Primary recommendation:** Build in this exact order — types → API client → hooks → chart installs → chart wrappers → skeletons → error boundaries. Each layer unblocks the next with no circular dependencies.

### Pattern 2: Shared API Client Utility

**What:** A single `apiRequest<T>()` function handles auth header injection, base URL construction, and error throwing. All feature-specific API functions call this utility.

**When to use:** Every API call to the FastAPI backend. Never call `fetch()` directly in hooks or components.

**Example:**
```typescript
// src/lib/api/client.ts
// Pattern: mirrors existing health.ts but generalized

export async function apiRequest<T>(
  path: string,
  token: string | null,
  options?: RequestInit
): Promise<T> {
  const url = `${process.env.NEXT_PUBLIC_API_URL}${path}`
  const response = await fetch(url, {
    ...options,
    headers: {
      'Content-Type': 'application/json',
      ...(token ? { Authorization: `Bearer ${token}` } : {}),
      ...options?.headers,
    },
    cache: 'no-store',
  })
  if (!response.ok) {
    throw new Error(`API error ${response.status}: ${path}`)
  }
  return response.json() as Promise<T>
}
```

### Pattern 3: React Query Hook Structure

**What:** Each domain has a hook in `src/hooks/` that calls `useAuth()` to get a Clerk token, then calls the domain API function via `useQuery`.

**When to use:** All data consumption in client components. Never call API functions directly in components.

**Example — useTransactions:**
```typescript
// src/hooks/use-transactions.ts
'use client'

import { useAuth } from '@clerk/nextjs'
import { useQuery, keepPreviousData } from '@tanstack/react-query'
import { fetchTransactions } from '@/lib/api/transactions'
import type { TransactionFilters, TransactionListResponse } from '@/types/transaction'

export function useTransactions(filters: TransactionFilters = {}, page: number = 0) {
  const { getToken } = useAuth()
  const limit = 25
  const offset = page * limit

  return useQuery<TransactionListResponse>({
    queryKey: ['transactions', filters, page],
    queryFn: async () => {
      const token = await getToken()
      return fetchTransactions(token, { ...filters, offset, limit })
    },
    placeholderData: keepPreviousData,   // keeps old page visible while fetching next
    staleTime: 30 * 1000,
  })
}
```

**Example — useDashboardSummary:**
```typescript
// src/hooks/use-dashboard-summary.ts
```

### Pattern 6: Error Boundary

**What:** A React class component (or `react-error-boundary` package) that catches rendering errors and displays a fallback UI instead of crashing the whole app.

**When to use:** Wrap every page client component and major section.

**Example — Custom implementation (if react-error-boundary not approved):**
```typescript
// src/components/shared/error-boundary.tsx
'use client'

import { Component, type ReactNode, type ErrorInfo } from 'react'

interface Props {
  children: ReactNode
  fallback?: ReactNode
}

interface State {
  hasError: boolean
  error?: Error
}

export class ErrorBoundary extends Component<Props, State> {
  constructor(props: Props) {
    super(props)
    this.state = { hasError: false }
  }

  static getDerivedStateFromError(error: Error): State {
    return { hasError: true, error }
  }

  componentDidCatch(error: Error, info: ErrorInfo) {
    console.error('ErrorBoundary caught:', error, info.componentStack)
  }

  render() {
    if (this.state.hasError) {
      return this.props.fallback ?? (
        <div className="flex flex-col items-center justify-center p-8 text-center">
          <p className="text-muted-foreground">Something went wrong. Please refresh the page.</p>
        </div>
      )
    }
    return this.props.children
  }
}
```

**Example — Using react-error-boundary package (if approved):**
```typescript
import { ErrorBoundary } from 'react-error-boundary'

function ErrorFallback({ error }: { error: Error }) {
  return <div>Something went wrong: {error.message}</div>
}

// Usage:
<ErrorBoundary FallbackComponent={ErrorFallback}>
  <DashboardPage />
</ErrorBoundary>
```

### Anti-Patterns to Avoid

- **Importing recharts directly:** Always consume the shadcn `ChartContainer` wrapper — importing `recharts` directly bypasses the OKLCH color token wiring.
- **Calling `useAuth()` in display components:** Only hooks in `src/hooks/` should call `useAuth()`. Components receive only typed data via props.
- **Prop-drilling tokens:** Never pass the Clerk token as a prop through component trees — it belongs only in React Query `queryFn` callbacks.
- **`amount` as `number`:** The backend `Decimal` type serializes as a JSON string. TypeScript interface must type `amount` as `string`, not `number`, to avoid silent truncation.
- **Calling `getToken()` outside `queryFn`:** `getToken()` is async and must be awaited inside `queryFn`, not in the hook body.
- **Module-scope `QueryClient`:** The existing `QueryProvider` already correctly uses `useState` — do not create additional `QueryClient` instances.
- **`date` as `Date` object in interfaces:** FastAPI serializes Python `datetime` as ISO 8601 strings. TypeScript types should use `string`, then parse with `new Date()` only at display time.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Chart rendering | Custom canvas/SVG charts | shadcn/ui Chart (Recharts) | Recharts handles responsiveness, tooltips, legends, accessibility |
| Auth token injection | Custom fetch wrapper with token storage | `useAuth().getToken()` in React Query `queryFn` | Clerk handles token refresh, expiry, and multi-session automatically |
| Data loading states | `isLoading` conditional plus manual state | React Query `isLoading`, `isPending`, `isError` | React Query handles all async state transitions correctly including background refetch |
| Cache invalidation | Manual re-fetch timers or setInterval | `queryClient.invalidateQueries()` + `staleTime` config | React Query cache is correct by default; manual timers drift and cause double-fetches |
| Error catching for async | try/catch in every component | React Query `isError` + `error` + `ErrorBoundary` | Two-layer strategy: React Query handles data fetch errors; ErrorBoundary handles render errors |
| Skeleton animation | CSS keyframes | shadcn `Skeleton` primitive | Uses Tailwind `animate-pulse` consistently; no animation drift between components |

**Key insight:** Every "custom solution" in this layer introduces a new contract that downstream pages must understand. The standard tools (React Query, shadcn primitives, Clerk hooks) are already installed, well-tested, and documented — use them verbatim.

## Common Pitfalls

### Pitfall 1: `amount` Typed as `number` Instead of `string`

**What goes wrong:** TypeScript shows no error, but JavaScript silently loses precision on large decimal values (e.g., `1234567.89` becomes `1234567.9`). Financial amounts must be rendered as-is from the API string.

**Why it happens:** Developers assume JSON numbers are safe for monetary values. The Python `Decimal` type serializes as a string in JSON, which must be preserved.

**How to avoid:** Type `amount` as `string` in `TransactionResponse`. Only convert to a displayable format using `Intl.NumberFormat('de-DE', { style: 'currency', currency: 'EUR' })` at render time. Never parse to `parseFloat()` for display.

**Warning signs:** Category edit modal shows rounded amounts; chart tooltips show truncated values.

---

### Pitfall 2: Charts Crash at Build Time (`next build`)

**What goes wrong:** Even with shadcn/ui Chart (Recharts), `recharts` internally uses `ResizeObserver` and DOM APIs that may cause hydration errors if the component tree is pre-rendered server-side with mismatched state.

**Why it happens:** Chart components that render SVG with dimensions derived from container measurements produce different values on server vs client.

**How to avoid:** Always mark chart wrapper files with `'use client'`. The `ChartContainer` from shadcn/ui handles the `ResizeObserver` usage correctly inside a client boundary. Run `npm run build` locally before declaring a chart component done — dev mode masks these errors.

**Warning signs:** Hydration error in console ("Text content did not match..."); charts render correctly in dev but break on `npm run build`.

---

### Pitfall 3: Missing `keepPreviousData` on Paginated Transactions

**What goes wrong:** Without `placeholderData: keepPreviousData`, the transaction table flashes empty/skeleton state on every page change, even when the new page is loading quickly.

**Why it happens:** React Query v5 defaults to clearing the previous data when the query key changes. Page transitions change the query key (includes `page` number).

**How to avoid:** Pass `placeholderData: keepPreviousData` (imported from `@tanstack/react-query`) to `useTransactions`. This keeps the old page visible until the new page arrives.

---

### Pitfall 4: staleTime Default Causes Double-Fetch on Navigation

**What goes wrong:** With `staleTime: 0` (the global default before any override), navigating away and back to a page causes React Query to immediately refetch even if data is seconds old.

**Why it happens:** The global `QueryProvider` already sets `staleTime: 60_000` as a default — but per-query overrides that set `staleTime: 0` (even implicitly by omitting it when the global is overridden) will trigger redundant fetches.

**How to avoid:** Never set `staleTime: 0` explicitly on queries. The global `staleTime: 60_000` in `QueryProvider` is the floor. For expensive endpoints (insights, analytics), set higher values: 5+ minutes.

---

### Pitfall 5: Error Boundary is a Class Component — Cannot Use Hooks Inside It

**What goes wrong:** Developers try to put `useQueryClient()` or `toast()` calls inside the `ErrorBoundary` render method or `componentDidCatch`. This throws a React error because hooks cannot be used in class components.

**Why it happens:** The mental model from functional components doesn't carry over. Error boundaries must be class components (this is a React constraint, not a framework choice).

**How to avoid:** Keep `ErrorBoundary` as a simple class component with no hooks. Wrap the error fallback UI in a separate functional component if it needs query client access. For toast notifications on error, use `componentDidCatch` with `sonner`'s `toast()` (not a hook — it's a regular function call).

---

### Pitfall 6: `useCategories` Makes an Unnecessary API Call

**What goes wrong:** Implementing `useCategories` as a React Query hook calling a non-existent `/categories` endpoint returns a 404 and wastes a network request on every page load.

**Why it happens:** Developers assume categories are dynamic data from the backend. They are actually a static Python enum.

**How to avoid:** Define `CATEGORY_OPTIONS` as a constant array derived from the `CategoryEnum` TypeScript type. The `useCategories` hook returns this static array with `isLoading: false`. No API call needed.

### Transactions API Function

```typescript
// src/lib/api/transactions.ts
import type { TransactionListResponse, TransactionFilters } from '@/types/transaction'
import { apiRequest } from './client'

export async function fetchTransactions(
  token: string | null,
  filters: TransactionFilters = {}
): Promise<TransactionListResponse> {
  const params = new URLSearchParams()
  Object.entries(filters).forEach(([key, value]) => {
    if (value !== undefined && value !== null) {
      params.set(key, String(value))
    }
  })
  const query = params.toString()
  return apiRequest<TransactionListResponse>(
    `/api/v1/transactions${query ? `?${query}` : ''}`,
    token
  )
}
```

## Open Questions

1. **react-error-boundary vs custom ErrorBoundary class**
   - What we know: STATE.md explicitly flags this as a blocker for Phase 1: "Confirm whether `react-error-boundary` npm package is acceptable to add, or whether a custom implementation is preferred."
   - What's unclear: User preference — package vs zero-dependency custom class.
   - Recommendation: Plan both implementations; default to custom class component (zero new dependency) unless user confirms package is fine. The custom implementation is ~40 lines and covers all requirements.

2. **`AnalyticsResponse.data` shape per endpoint**
   - What we know: The backend schema types `data` as `Dict[str, Any]` — intentionally generic. The actual nested shape (e.g., what keys are in `spending.data`, `categories.data`) is computed by the analytics service at runtime.
   - What's unclear: Whether the frontend needs strongly-typed sub-shapes or can tolerate `Record<string, unknown>` for Phase 1.
   - Recommendation: Type as `Record<string, unknown>` in Phase 1 (correct for Foundation layer). Dashboard and Analytics pages (Phases 2 and 4) can narrow the types when they know which fields to render.

3. **Insights regeneration mechanism**
   - What we know: The backend `GET /api/v1/insights` generates synchronously on first call with no separate POST trigger. The `InsightsResponse` schema does not include a `next_available_at` or cooldown field — the 1-hour cooldown described in requirements is frontend-enforced only.
   - What's unclear: How the `useInsights` hook should signal "generation in progress" if the first call takes 15-30 seconds.
   - Recommendation: The `isLoading` state from `useQuery` is sufficient — it will be `true` for the full duration of the synchronous generation. The skeleton shown during `isLoading` on the Insights page should display "Analyzing your spending patterns..." messaging. No polling. This is a Phase 5 concern — the hook itself is simple.
